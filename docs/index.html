1. Project description

Develop a multi-device (micro-)service-oriented system able to model and manage, in a smart way, public technical content regarding Web development (such as tutorials, presentations, examples of source-code, news, events, etc.) from multiple sources such as DevDocs, GitHub Pages, MDN Web Docs, Reddit, and others. Using the Linked Data principles, the application will generate and expose for human users and software - via a SPARQL endpoint - knowledge of interest for Web developers according to various criteria: topic (e.g., a certain programming language/paradigm, only guides/references about a specific framework), target platform, purpose, geographical area, period of time, user preferences,...

5. Linked Data Principles

Linked Data is a set of principles and best practices for structuring, connecting, and exposing data on the web in a way that allows it to be easily linked and interrelated. The core principles include:

  • Uniform Resource Identifiers (URIs): Every resource should be uniquely identified using a URI.
  • HTTP URIs: URIs should be accessible via HTTP, enabling dereferencing for retrieving resource representations.
  • Resource Representation: Resources should be represented in a standard format, usually using RDF (Resource Description Framework).
  • Links Between Resources: Include links in data to connect related resources, enabling navigation and discovery.
In the context of this project, Linked Data principles are applied to model technical content related to web development. Each piece of content, such as tutorials, examples, and documentation, is represented as a resource with a unique URI. Relationships between resources are established using links, creating a web of interconnected knowledge.


6. Retrieving Data

We use a web scraping service to get the data from the following sources: Reddit, MDN Web Docs, GitHub Topics.
We then process the retrieved data and create RDF triples in accordance with Linked Data principles.
These RDF triples represent the relationships and attributes of the collected data, allowing for seamless integration into our triplestore. Once the RDF triples are prepared, we load them into the selected triplestore, ensuring that our SPARQL endpoint is populated with the interconnected data.
After that, we can query the SPARQL endpoint to retrieve the data and present it to the user in the frontend of our application.


7. Ontology

We'll make use of the following ontology and vocabularies in order to model our business case.

  • PREFIX schema: https://schema.org/
  • PREFIX rdf: http://www.w3.org/1999/02/22-rdf-syntax-ns#
  • PREFIX rdfs: http://www.w3.org/2000/01/rdf-schema#
  • PREFIX xsd: http://www.w3.org/2001/XMLSchema#
  • PREFIX dc: http://purl.org/dc/elements/1.1/
  • PREFIX tl: http://purl.org/NET/c4dm/timeline.owl#
  • PREFIX event: http://purl.org/NET/c4dm/event.owl#
  • PREFIX foaf: http://xmlns.com/foaf/0.1/
  • PREFIX ns1: http://wade-project.org/news#
We will create an ontology that we can use to model the data we collect from the various sources. For example: 
ns1:NewsItem rdf:type rdf:Class ;
rdfs:label "News Item" .

ns1:link rdf:type rdf:Property ;
    rdfs:label "link" ;
    rdfs:domain ns1:NewsItem ;
    rdfs:range schema:URL .

ns1:origin rdf:type rdf:Property ;
    rdfs:label "origin" ;
    rdfs:domain ns1:NewsItem ;
    rdfs:range rdf:Literal .

ns1:title rdf:type rdf:Property ;
    rdfs:label "title" ;
    rdfs:domain ns1:NewsItem ;
    rdfs:range rdf:Literal .

ns1:topic rdf:type rdf:Property ;
    rdfs:label "topic" ;
    rdfs:domain ns1:NewsItem ;
    rdfs:range rdf:Literal .

# Examples:
ns1:CSS_Nesting rdf:type ns1:NewsItem ;
    ns1:link  ;
    ns1:origin "mozilla" ;
    ns1:title "CSS nesting" ;
    ns1:topic "tutorials" .

ns1:DOS_Wasm_X rdf:type ns1:NewsItem ;
    ns1:link  ;
    ns1:origin "reddit" ;
    ns1:title "DOS Wasm X: a new web based DOS Emulator" ;
    ns1:topic "javascript" .


8. User Manual

The user is able to login and register using the User Service.
After logging in, the user can access the preferences page, where they can select the topics they are interested in.

  • Each user has their own preferences, which are stored in the Preference Service and are used to filter the news items retrieved from the SPARQL endpoint.
  • Preferences can be changed at any time in order to refresh the feed.

When accessing the main page, users can see the latest news regarding the topics and preferences they have selected.
  • These news are retrieved from the SPARQL endpoint and displayed in the frontend of the application.
  • The user can also access the news from the sources they were retrieved from, by clicking on the link provided in the news item.
